# Task4 光伏发电预测 - 消融实验总结

## 实验概述

本文档记录了 Task4（720h→168h 光伏功率预测）项目中尝试过但效果不佳的优化方案，作为消融实验的完整记录，为后续研究提供参考。

**基准配置**：
- 输入：720小时（30天）历史数据
- 输出：168小时（7天）功率预测
- 基准模型：Mamba + MSE损失
- 基准指标：R² = 0.519，MLEF集成 R² = 0.558

---

## 1. 峰值损失函数实验

### 1.1 CombinedPeakLoss（失败）

**目标**：解决 Mamba 峰值低估 24.7% 的问题

**配置**：
```python
peak_loss_config = {
    'alpha': 1.0,      # 整体曲线 MSE
    'beta': 2.5,       # 峰值大小权重
    'gamma': 1.5,      # 峰值时刻权重
    'delta': 0.5,      # 峰值形状权重
    'use_soft_peak': True,
}
```

**结果**：
- 训练在 epoch 13 发散，loss 从 0.02 跳到 0.15
- 调整为保守权重（β=0.5, γ=0.3）后仍然不稳定

**失败原因**：
1. 峰值损失的梯度与 MSE 梯度量级差异大，导致优化不稳定
2. 软峰值提取（softmax）在训练早期不够稳定
3. 多个损失项的权重难以平衡

**教训**：
- 复杂的组合损失函数需要更细致的梯度裁剪和学习率调度
- 建议先单独验证每个损失项的效果

---

### 1.2 SoftPeakAwareLoss（失败）

**目标**：简化版峰值损失，解决 CombinedPeakLoss 不稳定问题

**配置**：
```python
soft_loss_config = {
    'alpha': 1.0,      # MSE 权重
    'beta': 0.3,       # 峰值大小权重（降低）
    'gamma': 0.1,      # 峰值时刻权重（降低）
    'delta': 0.0,      # 禁用形状损失
    'temperature': 2.0 # 软化 softmax
}
```

**结果**：
| 指标 | MSE基准 | SoftPeakAwareLoss |
|------|---------|-------------------|
| R² | 0.479 | 0.17 |
| 峰值偏差 | -24.7% | -33.87% |

**失败原因**：
1. 温度参数过高（2.0）导致峰值信号过度平滑
2. 低权重（β=0.3）不足以引导模型关注峰值
3. 软化的 softmax 削弱了峰值梯度信号

**教训**：
- 软化策略需要与权重调整配合
- 峰值损失可能不适合端到端训练，更适合作为后处理

## 2. 特征工程实验

### 2.1 改进峰值特征（71维 vs 3375维）（失败）

**目标**：用更精简、更有针对性的特征替代原始统计特征

**新特征设计（71维）**：
```python
features = [
    # 近期统计（7天）- 28维
    last_7d_mean, last_7d_std, last_7d_max, last_7d_min,  # ×7天

    # 趋势特征 - 14维
    diff_1d, diff_3d,  # ×7天

    # 周期特征 - 21维
    same_weekday_mean, same_weekday_std, same_weekday_max,  # ×7天

    # 时间编码 - 8维
    month_sin, month_cos, weekday_sin, weekday_cos, ...
]
```

**原始特征（3375维）**：
```python
# 720小时 × 25特征 的统计摘要
features = [mean, std, max, min, ...] × 每个特征 × 多个时间窗口
```

**结果**：
| 指标 | 原始3375维 | 改进71维 |
|------|-----------|----------|
| 峰值 R² | -0.251 | -0.490 |
| 峰值 MAE | 28.5 kW | 32.1 kW |
| 时刻 MAE | 2.5h | 4.3h |

**失败原因**：
1. 精简特征丢失了长期模式信息
2. 7天窗口太短，无法捕捉月度/季节变化
3. 原始高维特征虽然冗余，但包含更多潜在有用信号

**教训**：
- 特征工程应该是增量式的（添加新特征），而非替换式
- 高维统计特征对树模型有效，LightGBM/XGBoost 有内置特征选择

---

### 2.2 峰值时刻回归 vs 分类（失败）

**目标**：将24类分类改为回归任务，获得连续的时刻预测

**配置**：
```python
# 分类任务（基准）
y_hour = np.argmax(daily_power)  # 0-23

# 回归任务（实验）
y_hour_continuous = np.argmax(daily_power).astype(float)  # 0.0-23.0
```

**结果**：
| 指标 | 24类分类 | 回归任务 |
|------|----------|----------|
| 准确率/MAE | 15.3% | 4.3h |
| ±1h准确率 | 45% | 16% |

**失败原因**：
1. 回归对异常值敏感，少数错误预测（如预测0点）会严重拉高 MAE
2. 时刻本质上是离散的，回归输出需要后处理（四舍五入）
3. 分类任务的交叉熵损失对多峰分布更鲁棒

**教训**：
- 时刻预测更适合分类或粗粒度分类（4类：上午/中午/下午/傍晚）
- 如果用回归，需要添加边界约束（如 6-18h 范围限制）

## 3. 集成策略实验

### 3.1 残差校正器 PeakResidualCorrector（失败）

**目标**：训练后处理模型校正峰值预测残差

**架构**：
```python
# 特征（每天4维）
features = [
    pred_peak_value,      # 预测峰值大小
    pred_peak_hour,       # 预测峰值时刻
    model_disagreement,   # 模型间分歧度（std）
    peak_sharpness        # 峰值尖锐度（二阶导）
]

# 模型
correctors = [LightGBM() for _ in range(7)]  # 7天各一个
```

**结果**：
| 指标 | 无校正 | 残差校正 |
|------|--------|----------|
| 验证集 MAE | 39 kW | 24 kW ✓ |
| 测试集 R² | 0.558 | 0.510 ✗ |

**失败原因**：
1. 验证集过拟合：每天独立训练，样本量不足（~1700/天）
2. 特征与残差的相关性在测试集上不成立
3. 校正器放大了预测误差而非修正

**教训**：
- 后处理模型需要独立的验证集评估
- 应使用单一模型预测所有天的残差，增加样本量
- 添加正则化或减少特征数量

---

### 3.2 分时段权重 PeriodBasedEnsemble（无提升）

**目标**：夜间/白天/峰值时刻使用不同的模型权重

**配置**：
```python
periods = {
    'night': (0-5h, 19-23h),   # 夜间
    'daytime': (6-9h, 15-18h), # 白天非峰值
    'peak': (10-14h)           # 峰值时刻
}
temperature = 0.1  # softmax 温度
```

**结果**：
| 方法 | R² | 较MLEF |
|------|-----|--------|
| MLEF (基准) | 0.558 | - |
| Period_Based | 0.551 | -1.3% |
| Combined_Period_Time | 0.552 | -1.1% |

**未提升原因**：
1. MLEF 的动态置信度已经隐式学习了时段差异
2. 各模型在不同时段的 RMSE 差异不大（std=0.003）
3. 硬编码的时段划分不如数据驱动的动态权重灵活

**教训**：
- 当基准方法已经很强时，简单的改进难以超越
- 分时段权重更适合模型间差异明显的场景

---

### 3.3 增强峰值集成 EnhancedPeakAwareEnsemble（无提升）

**目标**：在峰值窗口内提高树模型权重

**配置**：
```python
peak_window = 2  # ±2小时
soft_transition_sigma = 1.0  # 高斯平滑
```

**结果**：
| 方法 | R² | 峰值RMSE |
|------|-----|----------|
| MLEF | 0.558 | 54.46 |
| Enhanced_MLEF | 0.549 | 54.14 |

**未提升原因**：
1. 树模型在峰值窗口的权重已经较高（0.21-0.25）
2. 峰值预测 R² 为负，增加权重反而引入更多误差
3. 软过渡边界模糊了峰值/非峰值的区分

## 4. 模型架构实验

### 4.1 Transformer（表现不佳）

**目标**：验证 Transformer 在超长序列任务的表现

**配置**：
```yaml
transformer_seq2seq_long:
  d_model: 128
  n_heads: 8
  num_layers: 4
  dropout: 0.2
  learning_rate: 0.0001
  batch_size: 16
```

**结果**：
| 指标 | Transformer | Mamba | 差距 |
|------|-------------|-------|------|
| R² | 0.462 | 0.512 | -9.8% |
| RMSE | 37.13 | 35.37 | +5.0% |
| 不确定性 | 38.78 kW | 31.39 kW | +23.5% |
| MLEF权重 | 0.138 | 0.171 | 最低 |

**表现不佳原因**：
1. 自注意力 O(n²) 复杂度：720×720 = 518,400 次注意力计算
2. 位置编码退化：标准位置编码对超长序列效果差
3. 显存限制：batch_size 只能用 16，训练效率低
4. 训练不充分：150 epoch 可能不够

**教训**：
- 超长序列任务（>500步）Mamba 优于 Transformer
- 如需使用 Transformer，考虑 Informer/Autoformer 等高效变体
- 或使用分段注意力、稀疏注意力降低复杂度

## 6. 关键结论

### 6.1 有效的方法

1. **Mamba + MSE**：长序列建模最优，O(n) 复杂度
2. **原始3375维统计特征**：虽冗余但信息完整
3. **MLEF 动态置信度加权**：自动学习最优组合
4. **树模型峰值融合**：校正峰值偏差（-19.6% → -1.5%）

### 6.2 无效的方法

1. **峰值感知损失**：训练不稳定或效果更差
2. **精简特征工程**：丢失关键信息
3. **分时段/残差校正**：无法超越基准
4. **Transformer**：不适合超长序列

### 6.3 建议的优化方向

1. **数据层面**：引入天气预报特征（辐照度、云量）
2. **模型层面**：尝试 Informer/PatchTST 等高效 Transformer
3. **损失函数**：分阶段训练（先 MSE 收敛，再微调峰值）
4. **集成层面**：基于功率水平的自适应集成
